## 2026-07-04 — Connector links redesigned as transparent, unbounded entry buffers

- **Goal:** minimum-impact connectors. Replaced the old finite connector (free-flow one-cell but with a fixed ~4-vehicle buffer and `4×` reference-capacity sizing) with a **transparent buffer**: `length = 1`, `v_f = w = 1/dt` (`T1 == T2 == 1`), and `rho_jam = 2·N` where `N` = the scenario's total vehicle count. With this triangular FD, storage = `2·N` vehicles and the per-step capacity token = `N`, so neither ever binds (verified numerically across `dt`/`N`).
- **Effect:** the whole entry queue for a junction origin now lives on its source connector — the origin can always offload every waiting vehicle (no double queueing at origin *and* connector), and queued vehicles are already in the connector's demand, so they discharge the instant the downstream junction's supply/priority allows (no extra origin→connector hop delay). A vehicle entering an empty connector still crosses in one free-flow step. The connector is otherwise served like any 1-lane link subject to downstream supply/merge priorities. The `N`-based budget has no `dt` term, removing the old buffer's `dt`-dependent spurious-delay risk.
- **Files:** `core/connector_link.py` (new `vehicle_budget` arg; dropped `reference_capacity`/`buffer_vehicles`); `network/network.py` (compute `vehicle_budget = sum of origin vehicle counts`, pass to both source/sink connectors; removed the now-dead `ref_cap` sizing and the `_fd` helper it used); `metrics/trips.py` (network entry = first *real* link, so waiting on an entry connector counts as `queue_delay`, not in-network `travel_time`). `_alpha_for` already weights a connector like the strongest *real* approach, so the inflated capacity does **not** leak into merge priorities; routing uses free-flow time, not capacity, so it is unaffected.
- **Verification:** congested junction-origin scenario — 30/30 arrive, origin vertical queue stays 0, connector holds the full queue. All gates green: 27 tests, pylint 10.00/10, black, mypy, 6 examples + CLI.
